# Data Model: Foundation for Future Integrations

## 1. WorkerScheduleOptions

- **Purpose**: Defines the recurring execution cadence required by the background service.
- **Fields**:
  - `IntervalHours`: required positive integer or equivalent duration representation stored in configuration
- **Validation Rules**:
  - Must be present at startup
  - Must be greater than zero
  - Invalid or missing values must fail configuration validation before normal recurring execution begins
- **Relationships**:
  - Consumed by `Worker`
  - Registered and validated in `Program`

## 2. OrchestrationRequest

- **Purpose**: Represents the implicit execution trigger sent by the worker to the application orchestrator for each cycle.
- **Fields**:
  - `TriggeredAt`: timestamp of the execution attempt
  - `CancellationToken`: cooperative cancellation signal passed through the call chain
- **Validation Rules**:
  - `TriggeredAt` is generated by the worker at execution time
  - `CancellationToken` must be forwarded unchanged
- **Relationships**:
  - Created by `Worker`
  - Handed to `FaturamentoOrchestrator`

## 3. Observability Signal Set

- **Purpose**: Defines the baseline operational signals the service must expose structurally.
- **Fields**:
  - `StartupLogEvent`
  - `ConfigurationFailureLogEvent`
  - `ExecutionCycleLogEvent`
  - `MetricNames`
  - `TraceActivityNames`
- **Validation Rules**:
  - Signal names must be stable and human-readable
  - Startup, schedule validation, and execution loop coverage must all be represented
- **Relationships**:
  - Emitted or reserved by `Program` and `Worker`
  - Extended later by infrastructure integrations

## 4. ServiceStructureMap

- **Purpose**: Captures the reserved logical boundaries for future integrations.
- **Fields**:
  - `ApplicationPath`
  - `DomainModelsPath`
  - `DomainInterfacesPath`
  - `InfrastructureStoragePath`
  - `InfrastructureAutomationPath`
  - `InfrastructureEmailPath`
- **Validation Rules**:
  - All reserved paths must exist in the project baseline
  - No area-specific adapter implementation is required in this increment
- **Relationships**:
  - Referenced by maintainers as the canonical placement map for later work

## State Notes

- There is no business entity lifecycle yet because invoice processing behavior is intentionally out of scope.
- The only active lifecycle in this increment is configuration validation followed by recurring worker execution readiness.
